## Workflow A：初始化标注

**触发场景**：用户提供PRD内容和原型HTML，要求开始标注。

### 第一步：获取数据

从 Notion「PM需求库」读取对应需求页面，获取：
- 功能列表JSON（包含所有Feature ID、名称、描述）
- PRD内容（提取每个功能点的验收标准摘要）
- 原型链接（GitHub Pages地址）

同时从GitHub仓库读取原始原型HTML文件内容。

### 第二步：模块化需求聚合

标注前必须先对PRD需求点进行逻辑分组：

**组件归一化原则**：严禁在同一个组件或关联模块上标记多个角标，将属于同一功能区域的需求整合至一个编号及一个浮窗内。

**聚合示例**：
- **行操作**：将「编辑、删除、查看、权限控制」等所有行内操作整合，标注在列表行「操作」列标题处
- **筛选模块**：将所有输入框、下拉框、查询/重置按钮的需求整合，标注在筛选模块整体右上角
- **Tab/容器**：将所有标签页切换逻辑、权限逻辑、数据逻辑整合，标注在Tabs组件整体右上角

**零丢失原则**：整合后的浮窗必须包含该模块下所有原始描述、业务逻辑、前置条件及异常流程，**严禁概括或删减细节**。

### 第三步：标注定位规范

```css
.annotation-badge {
  position: absolute;
  top: -8px;
  right: -4px;
  display: inline-block;
  vertical-align: top;
  background: rgb(250, 173, 20);
  color: white;
  font-size: 10px;
  font-weight: bold;
  line-height: 14px;
  padding: 0px 4px;
  border-radius: 2px;
  border: none;
  cursor: pointer;
  z-index: 9998;
}
```

**层级管理**：
- 若目标组件存在 `overflow: hidden`，将角标挂载至 `body` 根节点并计算全局坐标
- 确保角标不被任何页面元素遮挡

### 第四步：浮窗全量信息架构

浮窗内容排版规范（**必须严格执行**）：

```
【顶部标题栏】
需求编号（与角标样式一致）+ 需求描述：[模块名称]
下方有一条浅灰色细分割线

【Markdown 深度渲染】
- 段落行高：1.6；段落间距 margin-bottom: 12px
- 1:1还原原文中的**加粗**和*斜体*
- 严格保留多级无序列表和有序列表的层级嵌套
- 引用块以浅灰色左边框形式展示

【核心板块】
采用小标题形式：显示样式 / 交互与排序 / 业务定义 / 备注等

【视觉增强】
涉及状态色时，在文字前加对应彩色圆点图标
```

**浮窗样式**：
```css
.annotation-popup {
  position: fixed;
  width: 450px;
  background: #f0efef;
  border-radius: 4px;
  box-shadow: 0 4px 16px rgba(0,0,0,0.15);
  z-index: 9999;
  cursor: move;
}
```

**浮窗交互规范**：
- **触发**：鼠标 Hover 角标即刻打开浮窗
- **唯一性**：同一编号浮窗只能同时打开一个；支持同时打开多个不同编号的浮窗
- **关闭机制**：只能通过右上角「X」关闭按钮关闭
- **自由拖拽**：浮窗支持鼠标点击并自由拖移
- **事件隔离**：点击浮窗内部及拖动时，必须阻止事件冒泡
- **智能避让**：浮窗默认出现在角标左下方（间距3px），靠近页面边缘时自动反向调整

### 第五步：双向写入

1. **反向注入**：生成的需求编号（1-999）**反向写入PRD对应需求描述起始位置**（格式：`[1]`）
2. **逻辑同步**：确保页面标注编号与文档内编号严格一一对应

### 第六步：推送到GitHub

将标注版原型HTML推送到GitHub仓库：
- **文件名**：`[原文件名]-annotated.html`（如 `product-target-management-annotated.html`）
- **commit message**：`feat: 新增[需求名称]原型标注`
- 更新 `index.html` 导航页，追加标注版原型入口

### 第七步：更新Notion

- 将标注版原型的GitHub Pages链接写入Notion需求页面
- 更新PRD内容（包含反向写入的编号）
- 更新状态为「标注完成」

### 第八步：强制自检

- [ ] **聚合自检**：同一组件/模块处是否只有一个角标？
- [ ] **完整性检查**：浮窗信息是否详实到可完全替代PRD文档？
- [ ] **交互自检**：浮窗是否支持拖拽？关闭按钮是否有效？事件是否隔离？
- [ ] **样式自检**：角标是否符合10px粗体规范？层级是否正确？
- [ ] **反向写入自检**：编号是否已正确写回PRD？
- [ ] **GitHub推送自检**：文件是否成功推送？链接是否可访问？

## Workflow B：标注内容更新

**触发场景**：用户提供调整后的PRD或页面，要求更新现有标注。

1. **差异识别**：对比现有标注与调整后PRD，识别「新增、修改、删除」变化
2. **样式锁定**：严禁修改视觉样式参数（颜色、尺寸、字号、偏移量等）
3. **精准内容替换**：
   - 删除项：移除对应编号的角标和浮窗
   - 新增项：按规范生成新编号，编号逻辑递增续接
   - 修改项：仅替换浮窗内容，不改动角标位置
4. **反向写入**：同步更新PRD中的需求编号标记
5. **推送GitHub**：推送更新后的标注版原型，commit message：`fix: 更新[需求名称]原型标注`
6. **同步Notion**：写回最新版本，更新状态
